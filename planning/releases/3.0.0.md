# modern-di-grpc 3.0.0 — modern-di 3.x

Requires **modern-di >= 3, < 4**. **No public API change** — `DIInterceptor`,
`DIAioInterceptor`, `FromDI`, `inject`, `fetch_di_container`, and
`grpc_context_provider` keep their signatures and behavior. This release
includes a real adapter bug fix required by
[modern-di 3.0's mandatory-open lifecycle](https://modern-di.modern-python.org/migration/to-3.x/):
the per-request child container was never opened.

## Fix

- **The per-request `REQUEST` child container is now opened.** `_build_child`
  built the child container for each RPC and returned it — the four
  interceptor wrappers (`_wrap_unary_sync`/`_wrap_stream_sync` →
  `close_sync`, `_wrap_unary_aio`/`_wrap_stream_aio` → `close_async`) only
  ever *closed* it, never opened it. Under modern-di 3.0's mandatory-open
  lifecycle this meant every RPC raised `ContainerClosedError` on its first
  resolve. `_build_child` now calls `child.open()` right after building the
  child, before returning it — one fix, since all four wrappers route
  through it.

## Packaging

- **Bumped the `modern-di` dependency to `>=3,<4`.** Requires modern-di 3.x;
  drops support for modern-di 2.x.

## Downstream

**You must now open the root container yourself before serving RPCs.** gRPC
gives `DIInterceptor`/`DIAioInterceptor` no server-startup hook to attach to,
so — the same way root teardown has always been the caller's responsibility
(`close_sync()`/`close_async()` after `server.stop(...)`) — opening it first
is too:

```python
container = Container(groups=[AppGroup], validate=True)
container.open()  # or: with container: ... — required under modern-di 3.x
server = grpc.server(..., interceptors=[DIInterceptor(container)])
```

Passing an unopened container will raise `ContainerClosedError` on the very
first RPC. If your own code builds child containers elsewhere, open them
with `with` (or `open()`) before resolving — see the
[modern-di 3.x migration guide](https://modern-di.modern-python.org/migration/to-3.x/).
