# Design: modern-di integration for gRPC

**Status:** design, pending maintainer approval
**Target:** new repo `modern-python/modern-di-grpc`, package `modern_di_grpc`

## Summary

Ship the official [modern-di](https://modern-di.modern-python.org) integration
for [gRPC](https://grpc.io) (the `grpcio` package), following the core repo's
[Writing an integration](../../../modern-di/docs/integrations/writing-integrations.md)
contract. gRPC's idiomatic extension point is a **`ServerInterceptor`**, so this
integration is the **interceptor + `@inject` decorator** shape (the "middleware
owns the child, decorator resolves" split, like `modern-di-starlette`/
`-faststream`). One module (`modern_di_grpc/main.py`) exposes `DIInterceptor`
(sync), `DIAioInterceptor` (async), `FromDI`, `inject`, `grpc_context_provider`,
and `fetch_di_container`. The interceptor re-wraps each RPC behavior to build a
per-RPC `Scope.REQUEST` child container, stash it in a `ContextVar`, and close it
after the call; `@inject` resolves `FromDI` params from that child. Both the sync
thread-pool server (`grpc.server`) and the async server (`grpc.aio.server`) are
supported.

## Motivation

gRPC is a widely-used RPC framework with steady enterprise demand — a new
entrypoint modern-di has not covered. It is the roadmap's remaining "Next up"
integration. Dishka has a grpcio integration; modern-di has none.

## Verified facts (grpcio 1.82 + modern-di 2.27, spiked end-to-end against a real server + client)

Every non-obvious decision below was proven against installed grpcio, not
inferred:

- **A `ServerInterceptor.intercept_service(continuation, handler_call_details)`
  can re-wrap the RPC behavior.** It calls `continuation(handler_call_details)`
  to get the original `RpcMethodHandler`, then returns a *new* handler built with
  `grpc.{unary_unary,unary_stream,stream_unary,stream_stream}_rpc_method_handler`
  wrapping a new behavior that carries the original `request_deserializer`/
  `response_serializer`. Proven for all four RPC types (sync) and for
  unary-unary + server-streaming (aio).
- **The per-RPC child container reaches the servicer method via a
  `ContextVar`.** The wrapped behavior builds the child, `ContextVar.set(child)`,
  runs the original behavior (the servicer method, same thread/task), then
  `close`s the child and `ContextVar.reset(token)` in `finally`. `@inject` reads
  the ContextVar. Verified end-to-end in both the sync thread-pool server and the
  aio server; the same `grpc.*_rpc_method_handler` factories work for both.
- **Server-streaming behaviors MUST be generators** (`yield from` sync /
  `async for ... yield` aio) so the child stays open *during* iteration and
  closes only after the last item — `return`ing the handler's generator would
  close the child before gRPC consumes it. Proven: exactly one child closed per
  RPC across all four types, including streaming.
- **`ServicerContext` is injectable** via `providers.ContextProvider(
  ServicerContext, scope=Scope.REQUEST)` seeded with `set_context` in the wrapper.
  Verified a factory resolved the live `ServicerContext` (`ctx is context`).
- **gRPC always calls a behavior as `(request, context)` positionally**, so
  `@inject` needs **no** bind-by-name and **no** `*args`/`**kwargs` guard (the
  celery/arq argument-corruption class cannot occur): the wrapper resolves DI
  params and calls `func(*args, **kwargs, **resolved)`. `functools.wraps` is safe
  (gRPC registers the servicer method by name and never introspects/unwraps its
  signature).
- **gRPC has no server start/stop callback** the interceptor can hook. The
  container is handed to the interceptor already open (modern-di containers are
  created open), and the user closes it after `server.stop()`. Root lifecycle is
  user-owned, matching `modern-di-flask` ("no app-shutdown hook").
- **Testing uses a real server on `127.0.0.1:0`** (an OS-assigned port) with a
  real client channel — no Docker. Generated proto stubs are committed under
  `tests/`. `protobuf` is a test-only dependency (the generated `_pb2` modules
  import it); the runtime package needs only `grpcio` + `modern-di`.

## Design

### Connection provider (kind → scope)

One connection kind — an RPC call → `REQUEST`:

```python
grpc_context_provider = providers.ContextProvider(ServicerContext, scope=Scope.REQUEST)
```

`ServicerContext` (from `grpc`, no protobuf import) exposes RPC metadata,
deadline, and peer. The protobuf **request `Message`** is deliberately **not**
exposed as a provider: binding it would add a `protobuf` runtime dependency, and
the request is already a servicer-method argument.

### Scope model

**Every RPC call opens one `Scope.REQUEST` child** — one RPC = one unit of work
(like an HTTP request or a broker message), uniform across all four RPC types.
For streaming-request RPCs (`stream_unary`, `stream_stream`) there is no single
request `Message`, but the child and `ServicerContext` context are the same as
for unary-request RPCs.

### Container handoff (a `ContextVar`)

```python
_request_container: ContextVar[Container] = ContextVar("modern_di_request_container")
```

The interceptor `set`s the per-RPC child before running the behavior and
`reset`s it (via the returned token) in `finally`. `@inject` and
`fetch_di_container` read it. A `ContextVar` is the only handoff: the interceptor
wraps the behavior, but the servicer method is a separate callable, and
`ServicerContext` is a C-extension object that cannot carry arbitrary attributes.

### `DIInterceptor` (sync) and `DIAioInterceptor` (async)

Both take the root container and implement `intercept_service`. For each of the
four handler kinds, the wrapper builds a `Scope.REQUEST` child, seeds
`ServicerContext`, sets the ContextVar, runs the original behavior, and closes
the child + resets the token in `finally`:

```python
class DIInterceptor(grpc.ServerInterceptor):
    def __init__(self, container: Container) -> None:
        self._container = container

    def intercept_service(self, continuation, handler_call_details):
        handler = continuation(handler_call_details)
        if handler is None:
            return None
        return _wrap_handler_sync(handler, self._container)
```

`_wrap_handler_sync` dispatches on which of `handler.unary_unary` /
`unary_stream` / `stream_unary` / `stream_stream` is set and returns a new
handler via the matching `grpc.*_rpc_method_handler` factory:

- **Response-unary** (`unary_unary`, `stream_unary`) — the wrapper `return`s the
  behavior's result inside a `try/finally` that closes the child.
- **Response-streaming** (`unary_stream`, `stream_stream`) — the wrapper is a
  **generator** that `yield from`s the behavior inside a `try/finally`, so the
  child lives for the whole stream.

`DIAioInterceptor(grpc.aio.ServerInterceptor)` mirrors it with `async def`
behaviors: response-unary wrappers `await` and `close_async`; response-streaming
wrappers are **async generators** (`async for item in behavior(...): yield item`)
closing with `close_async` in `finally`. It uses the same
`grpc.*_rpc_method_handler` factories.

A tiny per-kind builder avoids repeating the four bodies; the sync/aio split is
the only real duplication (sync `close_sync`/`yield from` vs async
`close_async`/`async for`).

### `fetch_di_container() -> Container`

```python
def fetch_di_container() -> Container:
    return _request_container.get()
```

Returns the current RPC's child container from the ContextVar — for code inside a
servicer that wants the container without `@inject`. Raises `LookupError` if
called outside an intercepted RPC (documented).

### `FromDI` marker + `inject` decorator

`FromDI` is an inert frozen/slotted `_FromDI` marker (the decorator-path shape).
`inject` parses `Annotated[T, FromDI(...)]` params and, because gRPC always
passes `(request, context)` positionally, simply appends the resolved DI params
as keywords:

```python
def inject(func):
    di_params = _parse_inject_params(func)
    if not di_params:
        return func

    is_async = inspect.iscoroutinefunction(func) or inspect.isasyncgenfunction(func)

    if inspect.isasyncgenfunction(func):
        @functools.wraps(func)
        async def async_gen_wrapper(*args, **kwargs):
            container = _request_container.get()
            resolved = {n: container.resolve_dependency(m.dependency) for n, m in di_params.items()}
            async for item in func(*args, **kwargs, **resolved):
                yield item
        return async_gen_wrapper

    if is_async:
        @functools.wraps(func)
        async def async_wrapper(*args, **kwargs):
            container = _request_container.get()
            resolved = {n: container.resolve_dependency(m.dependency) for n, m in di_params.items()}
            return await func(*args, **kwargs, **resolved)
        return async_wrapper

    @functools.wraps(func)
    def sync_wrapper(*args, **kwargs):
        container = _request_container.get()
        resolved = {n: container.resolve_dependency(m.dependency) for n, m in di_params.items()}
        return func(*args, **kwargs, **resolved)
    return sync_wrapper
```

`inject` produces a sync wrapper, an async wrapper, or an **async-generator**
wrapper to match the servicer method — a server-streaming method is a generator
(sync) or async generator (aio) and must be wrapped as one so gRPC still sees an
iterator. `resolve_dependency` dispatches provider-vs-type. `functools.wraps`
preserves the method name gRPC registers by. A method with no `FromDI` is
returned unchanged.

### Public API (`__all__`)

`DIInterceptor`, `DIAioInterceptor`, `FromDI`, `inject`, `fetch_di_container`,
`grpc_context_provider`.

## Non-goals

- **Root lifecycle wiring** — gRPC has no server start/stop hook; the user creates
  the container (open) and closes it after `server.stop()` (matches Flask).
- **Sync-generator server-streaming written as a coroutine that calls
  `context.write(...)`** — the interceptor's response-streaming wrapper assumes
  the idiomatic **(async) generator** servicer form (what the spike proved).
  Dishka additionally handles the `context.write` coroutine form; modern-di keeps
  the conservative generator-only contract and documents it.
- **Exposing the protobuf request `Message`** as a provider — would add a
  `protobuf` runtime dependency; the request is already a method argument.
- **Client-side interceptors** — this integration is server-side DI only.
- **A `setup_di(app, ...)` function** — gRPC's entry point is passing the
  interceptor to `grpc.server(..., interceptors=[...])`; the interceptor class is
  the setup mechanism (like Litestar's plugin realizing the contract its own way).
- Any runtime dependency beyond `grpcio` and `modern-di`.

## Testing

A real server on `127.0.0.1:0` + a real client channel, for **both** the sync
thread-pool server and the aio server; 100%-coverage gate (`just test-ci`). A
small committed proto (`tests/protos/greeter.proto`) and its generated
`greeter_pb2.py`/`greeter_pb2_grpc.py` define a `Greeter` service with all four
RPC types. `protobuf` and `grpcio-tools` are dev dependencies (stubs are
regenerated via a `just proto` recipe; committed so tests need only `grpcio` +
`protobuf`).

- `dependencies.py` — a `Group` with an APP-scoped `Factory`, a REQUEST-scoped
  `Factory` with a finalizer (to prove per-RPC child teardown), and one reading
  the injected `ServicerContext` (to prove context injection, e.g. `context.peer()`).
- `test_sync.py` — a sync server with `DIInterceptor`; each of the four RPC types
  invoked through `@inject` servicer methods resolves APP- and REQUEST-scoped
  values; the per-RPC child's finalizer runs after each call (including
  streaming); a method with no `FromDI` is a passthrough; `fetch_di_container()`
  returns the live child inside an RPC.
- `test_aio.py` — the same matrix against a `grpc.aio.server` with
  `DIAioInterceptor` (async servicer methods / async generators), asserting
  `close_async` teardown.
- `test_context.py` — a factory injected with `grpc_context_provider` reads the
  live `ServicerContext`; `fetch_di_container()` raises `LookupError` outside an RPC.

## Risk

- **R1 — grpcio API drift (pre-2.0 but long-stable).** `ServerInterceptor`,
  `grpc.aio.ServerInterceptor`, `intercept_service`, `RpcMethodHandler`'s four
  behavior attributes, and the `grpc.*_rpc_method_handler` factories are relied
  on — all stable across modern grpcio. *Mitigation:* pin `grpcio>=1.48,<2`. The
  implementation step installs the low bound on Python 3.10, runs the suite, and
  raises the floor to the lowest green version if an API landed later (the aiogram
  3.0→3.2 / Flask 2→3 pattern).
- **R2 — ContextVar isolation under the sync thread pool.** Each RPC runs in a
  worker thread with its own context; the wrapper `set`s and `reset`s the
  ContextVar within one call on one thread, so concurrent RPCs cannot see each
  other's child. Proven by the shipped Dishka integration using the identical
  mechanism, and by the spike. A test issues concurrent RPCs to guard it.
- **R3 — Early child close on streaming.** Mitigated by the generator wrappers
  (R-tested: finalizer runs after the last streamed item, not before).
