# Change: Fix flaky 100%-coverage CI failure in test_aio.py

**Lane:** lightweight — 1 file, ~4 LOC, no public-API change, no new test.

## Goal

`checks / pytest (3.11)` was intermittently failing CI's `--cov-fail-under=100`
gate (~1-in-4 runs), always on the same two lines in
`test_aio_app_finalizer_runs_on_root_close`, even though all 27 tests passed.
Stop the flake without weakening the coverage gate.

## Approach

Root-caused via live reproduction on GitHub Actions (reran the workflow
against the same commit repeatedly; reproduced on attempt 7 of 8 total, same
two lines every time, only on the 3.11 job) plus a matching upstream report:
[coveragepy#2124](https://github.com/coveragepy/coveragepy/issues/2124)
documents coverage.py losing the lines immediately after awaiting a coroutine
that internally cancels a task and catches `CancelledError`, specific to
Python 3.11's frame handling. `server.stop(0)` does exactly that (it cancels
the server's own background tasks), so the two lines right after it in the
test were the tracer's blind spot. No other aio test has code after
`await server.stop(0)`, which is why only this one test ever showed the gap.

First attempt: inserted `await asyncio.sleep(0)` after `server.stop(0)` as a
throwaway scheduling checkpoint, hoping the tracer's blind spot would land
there instead. Reproduced on CI again immediately — the blind spot just
moved onto the checkpoint line itself, and the original failure had shown a
2-line miss (not 1), so the width isn't fixed at one line. Shifting the gap
doesn't fix it: `--cov-fail-under=100` fails on *any* missed line, wherever
it lands.

The actual fix: mark the two affected lines (`await container.close_async()`
and the assert) with `# pragma: no cover`. Coverage.py honors this via static
source parsing, not runtime tracing, so it removes those lines from the
statement count entirely — immune to the race regardless of how many lines
it swallows on a given run. Confirmed locally: `tests/test_aio.py`'s tracked
statement count drops from 107 to 105. See
[decisions/2026-07-13-coveragepy-2124-flake.md](../decisions/2026-07-13-coveragepy-2124-flake.md)
for the alternatives considered and why this one was chosen.

## Files

  explaining the non-obvious why.
