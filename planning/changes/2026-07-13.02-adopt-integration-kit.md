# Design: Adopt the modern-di integration kit

## Summary

`modern-di` 2.28.0 shipped `modern_di.integrations` — the framework-agnostic
primitives that formalize what this package's `_build_child` and
`FromDI`/`_parse_inject_params`/`_resolve` already hand-roll. This change
swaps the hand-rolled internals for kit calls. Public API and runtime
behavior are unchanged; every existing test asserts on public behavior
only.

## Motivation

Eighth of the 13 adapter conversions surveyed by the kit's own design — see
[modern-di's decision record](https://github.com/modern-python/modern-di/blob/main/planning/decisions/2026-07-13-integration-kit-shape.md),
whose 2026-07-13 addendum names this repo explicitly: "grpc's `_build_child`
collapses to one `bind()` call, dropping its post-hoc `set_context`
entirely." Unlike aiohttp (two connection providers sharing one type,
needing a hand-rolled probe), gRPC has exactly **one** connection provider
(`grpc_context_provider`) and no dispatch to do — so this conversion uses
`bind()` directly, with no `classify_connection` call anywhere (there being
only one provider, nothing to classify across). This mirrors
`modern-di-flask`'s/`modern-di-faststream`'s single-provider shape for
Layer 1.

For Layer 2, `_FromDI`/`_parse_inject_params`/`_resolve` map onto
`Marker`/`parse_markers`/`resolve_markers` byte-for-byte — the same triad
already validated in the
[`modern-di-starlette`](https://github.com/modern-python/modern-di-starlette/pull/8)
conversion. `_parse_inject_params`'s current body is already an exact
duplicate of what `integrations.parse_markers` does, so it is deleted
outright rather than kept as a pass-through wrapper (confirmed no test
imports it directly — only `_build_child`, `_request_container`, and
`_wrap_unary_sync` are imported white-box in `tests/test_inject.py`).

## Design

In `modern_di_grpc/main.py`:

- Import `integrations` from `modern_di`:
  `from modern_di import Container, Scope, integrations, providers`.
- `_build_child`: replace
  `child = container.build_child_container(scope=Scope.REQUEST); child.set_context(ServicerContext, context); return child`
  with `match = integrations.bind(grpc_context_provider, context); return container.build_child_container(scope=match.scope, context=match.context)`.
  `Scope` stays imported — the module-level `grpc_context_provider =
  providers.ContextProvider(ServicerContext, scope=Scope.REQUEST)`
  declaration is unaffected.
- Delete `_FromDI`; replace `FromDI`'s body with a direct assignment:
  `FromDI = integrations.from_di`.
- Delete `_parse_inject_params` entirely; `inject` calls
  `integrations.parse_markers(func)` directly where it used to call the
  deleted function.
- `_resolve` keeps its name and signature shape but its body becomes
  `return integrations.resolve_markers(container, di_params)` instead of
  the manual dict comprehension.
- Drop now-unused imports/names: `dataclasses` (only `_FromDI` used it),
  and both `T`/`T_co` TypeVars — unlike the `modern-di-typer`/
  `modern-di-flask` conversions, `inject`'s own signature here is typed
  `typing.Callable[..., typing.Any]` throughout (never generic over `T`),
  so once `_FromDI`/old `FromDI` are gone, neither TypeVar has a remaining
  use.

`__init__.py`'s re-exports are untouched — every public name keeps its
exact signature and behavior. No test constructs `_FromDI` directly
(verified by reading all test files), so no test-file edit is expected —
including `test_build_child_seeds_servicer_context`, which asserts on
`_build_child`'s externally observable behavior (`child.scope is
Scope.REQUEST`, `child.resolve(ServicerContext) is context`), both
preserved exactly by the `bind()` collapse.

## Non-goals

- `classify_connection` — gRPC has exactly one connection provider; a
  dispatch-across-providers function has nothing to dispatch across here
  (confirmed by the architecture doc's own description of
  `grpc_context_provider` as "the gRPC equivalent of a web integration's
  'connection provider'" — singular).
- Any change to `_rewrap`, `_wrap_unary_sync`/`_wrap_stream_sync`/
  `_wrap_unary_aio`/`_wrap_stream_aio`, `DIInterceptor`/`DIAioInterceptor`,
  `_ensure_context_provider`, or the `_request_container` ContextVar
  handoff mechanism — none of that is part of the kit; the wrapper
  builders keep calling `_build_child` exactly as before, just with its
  new internals.
- Any test rewrite. The existing suite asserts on public behavior only
  plus the two white-box internals named above — none of it references
  `_FromDI`, `_parse_inject_params`, or the manual `resolve_dependency`
  dict comprehension being replaced.

## Testing

- `just test-ci` — 100% line coverage, all existing tests green with zero
  test-file changes.
- `just lint-ci` — ruff (`select=ALL`), `ty check`, `check-planning`.

## Risk

- **Version-floor bump breaks on an environment still resolving `<2.28`**
  (low likelihood — semver-compatible; low impact — `pyproject.toml` pins
  the floor explicitly, in this repo's existing `>=2.25,<3` style without a
  patch component, kept consistent as `>=2.28,<3`).
- **Passing `context=` to `build_child_container` at construction time
  behaves differently from the post-hoc `child.set_context(...)` call**
  (low likelihood — `build_child_container(context=...)` is the same
  context-seeding path every other converted integration already relies on
  for its connection object, and `test_build_child_seeds_servicer_context`
  exercises exactly this: `child.resolve(ServicerContext) is context`
  after `_build_child` returns — this test is unmodified and must still
  pass for the task to be considered done).
